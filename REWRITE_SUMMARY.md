# SDSM HybridSDSM v2 — Rewrite Summary

This rewrite adds a new `HybridSDSM_v2` dissemination pipeline alongside the
existing four baselines (`Periodic`, `Greedy`, `GreedyBSMImplied`, legacy
`HybridSDSM`). v1 code paths are preserved byte-for-byte; the two paths are
selected at runtime by the new `hybridVariant` NED parameter (`"v1"` / `"v2"`).

## New CSV streams

All three are written by `HybridSDSM_v2` runs; v1 runs still create them but
only with headers (empty body), so nothing breaks.

- `*-triggers.csv` — columns: `time,node,selfChange_norm,objSetChange_norm,timeSinceSend_norm,cbr,triggered_terms,sent,reason`. One row per tick per vehicle (regardless of whether a packet was sent), so sensitivity sweeps can replay the trigger decision offline.
- `*-object-aoi.csv` — columns: `time,receiver,object_id,aoi`. 100 ms cadence per receiver × known ego-object.
- Extra columns in `*-vehicle-summary.csv`: `avg_object_aoi,p95_object_aoi,assoc_parsed,assoc_predicted,assoc_pass_coarse,assoc_pass_fine,assoc_matched,assoc_unmatched,assoc_heading_fallback`.
- Extra columns in `*-summary.csv`: `avg_object_aoi,p95_object_aoi,p99_object_aoi,assoc_parsed,assoc_predicted,assoc_pass_coarse,assoc_pass_fine,assoc_matched,assoc_unmatched,assoc_heading_fallback`.

2. **Spatial association** — counters are monotonically non-increasing across stages: parsed 10 914 015 → predicted 10 912 893 → passCoarse 7 219 508 → passFine 5 279 700 → matched 5 120 715. 3. **Object selection** — EV_INFO logs show candidate → afterConf → afterRedundancy → afterVoi → taken funnel per TX; v2 branch runs instead of the legacy VoI path.
4. **Adaptive frequency** — `-triggers.csv` distribution (82 k ticks): `hybrid_obj` 81 670, `hybrid_none` 1005, `hybrid_multi` 926, `hybrid_self` 374, `hybrid_time` 1. No `hybrid_suppressed`/`hybrid_backstop` seen at this density (CBR stayed near zero); the suppressor/backstop paths will exercise in denser scenarios.

## Simplifications vs. brief (flagged per Part 8)

1. **Brief internal inconsistency on K**: Part 1 specified `obj_measurement_time_ms[16]`, Part 3 said "K_max = 32 … update the 16-slot arrays in SdsmPayload.msg to 32-slot arrays." Resolved per your clarification to use **[32] everywhere** + `SDSM_PER_OBJECT_BYTES = 26`. Max packet body = 40 + 32·26 = 872 bytes.
2. **"Local object IDs"**: the simulator uses globally-unique SUMO node indices, but per your instruction the spatial-association pipeline treats them as if they were per-ego-local. `egoObjectId` = the ego's `neighborInfo_` key selected by the GNN. This is what the Allig / Wanielik / Volk stack would do on real hardware; IDs happen to be numerically equal in-sim but are only used as opaque handles.
3. **`obj_measurement_time_ms` wrap-around**: the field is `uint16_t`, which wraps every ≈65.5 s. Both sender and receiver derive from the same `simTime()`, so the low-16 arithmetic `dt_ms = (nowMs - fieldMs) & 0xFFFF` recovers the correct `dt` whenever `dt < 65.5 s` (always true since measurement happens within `detectionMaxAge_ = 2 s` of packing). Documented inline in `runSpatialAssociation` Stage 1.
4. **Heading-unavailable Mahalanobis velocity term**: the brief left this unspecified when `obj_heading == 28800`. Implementation uses an **axis-split approximation** `(vx, vy) = (vel/√2, vel/√2)` for both sides of the Mahalanobis term, so the Δv terms collapse sensibly. Per your ask, the `assoc_heading_fallback` counter tracks how often this fires and is surfaced in the run summary and per-vehicle summary so the approximation's impact can be audited. In the smoke run it fired 0.01% of the time.
5. **`computeObjectSetChange` reuse for `objSetChange_norm`**: the brief specified `(Σ kinematic deltas + 5·(new + lost IDs)) / (K_max · ref_dist)`. The existing `computeObjectSetChange()` (v1) already computes the numerator exactly, so `evaluateHybridSendV2` normalizes its output rather than duplicating the loop. No behavior change vs. the brief formula.
6. **Part 6 ambiguity re. preserving `[Config HybridSDSM]`**: per your clarification we kept **both code paths**. `[Config HybridSDSM]` still runs the v1 utility logic (`hybridVariant = "v1"`); `[Config HybridSDSM_v2]` sets `hybridVariant = "v2"` and K_max = 32. The v1 `computeVoiScore` and `evaluateHybridSend` are untouched.
7. **v1 runs also emit empty `-triggers.csv` / `-object-aoi.csv`**: the files are created by `openCsvLogs` for all runs (simpler lifecycle), but only v2 writes rows. v1 files contain only the header line and ≈40 bytes total. No analysis impact; trivially filter by row count.

## Addendum: Greedy-family v2 baselines (scheduler consistency)

After the initial rewrite shipped, the `HybridSDSM_v2` scheduler differed from the `Greedy` scheduler in *two* ways simultaneously: object selection (VoI + LARM + spatial association) *and* trigger logic (ETSI parallel-threshold OR vs. weighted-sum utility). That coupling meant the paper's "HybridSDSM_v2 vs. Greedy" delta conflated scheduler-level and object-selection-level effects. To isolate the scheduler change, the v2 parallel-threshold scheduler is now also available for the Greedy-family baselines via a new `greedyVariant` NED parameter.

### Files changed (addendum)

| File | Change |
|---|---|
| `src/RosSDSMApp.h` | Added `std::string greedyVariant_`. Renamed `evaluateHybridSendV2()` into a shared `evaluateV2Schedule(const char* algoTag)` with two thin wrappers (`evaluateHybridSendV2` → `"hybrid"` tag, `evaluateGreedyV2` → `"greedy"` tag). |
| `src/RosSDSMApp.cc` | Read `greedyVariant_` from NED; extended the `maxObjectsPerSdsm_` K_max override, `objectAoiTimer_` creation gate, and the sender-side `objectLastUpdate_` refresh in `onBSM` to fire when *any* v2 scheduler is active. `handleSelfMsg` now dispatches `greedyEnabled_ && greedyVariant_=="v2"` through `evaluateGreedyV2`. The RX spatial association pipeline (`runSpatialAssociation`) is still gated on HybridSDSM_v2 only — Greedy-family v2 variants keep distance-topK object selection so the comparison with HybridSDSM_v2 attributes cleanly to VoI+LARM. |
| `simulations/apps/RosSDSMApp.ned` | Added `string greedyVariant = default("v1")`. |
| `simulations/omnetpp.ini` | Added default `greedyVariant = "v1"` in the general section; added three new configs: `[Config Greedy_v2]`, `[Config EventTriggered_v2]`, `[Config GreedyBSMImplied_v2]`. |
| `run_experiments.py` | Added the three new algorithm names to `ALGORITHMS`. |

### Semantic summary

Each v2 Greedy-family config extends the same parallel-threshold scheduler (ETSI TS 103 324 §6.1 parallel triggers + TS 102 687 DCC suppressor + backstop) as `HybridSDSM_v2`:

| Config | Scheduler | Object selection | Payload | Trigger reasons in `-triggers.csv` |
|---|---|---|---|---|
| `HybridSDSM` (v1) | legacy weighted-sum | legacy VoI (distance/age/relSpeed) | full list, K≤16 | `hybrid_*` (v1 labels) |
| `HybridSDSM_v2` | parallel-threshold | VoI (Lyu et al.) + LARM + spatial association | full list, K≤32 | `hybrid_{self,obj,time,multi,backstop,suppressed,none}` |
| `Greedy` (v1) | legacy weighted-sum | distance top-K | full list, K≤16 | none (no `-triggers.csv` rows) |
| **`Greedy_v2`** | parallel-threshold | distance top-K | full list, K≤32 | `greedy_{self,obj,time,multi,backstop,suppressed,none}` |
| **`EventTriggered_v2`** | parallel-threshold with `tau_time=1e9`, `tau_cbr=1.01`, `T_max=1e6s` → only self/obj triggers active | distance top-K | full list, K≤32 | `greedy_{self,obj,multi,none}` only (no `time`/`backstop`/`suppressed`) |
| **`GreedyBSMImplied_v2`** | parallel-threshold | n/a (`bsmImpliedMode=true`, `numObjects=0`) | header-only | `greedy_*` |

Reason labels are prefixed by algoTag (`hybrid_` or `greedy_`) so rows remain disambiguated in merged CSV analyses; each config additionally writes to its own `logPrefix`-scoped filename (`Greedy_v2-r0-triggers.csv`, etc.), so no file collisions are possible in a shared `results/` directory.

### Smoke run verification (addendum)

- `Greedy_v2` (20 s): scheduler fires `greedy_self / greedy_obj / greedy_multi / greedy_suppressed / greedy_none` as expected. Association counters are 0 in summary (spatial association correctly *not* run). Per-object AoI is populated (`avg_object_aoi ≈ 1.72 s, p95 ≈ 7.85 s, p99 ≈ 12.1 s`).
- `EventTriggered_v2` (20 s): trigger reasons include only `greedy_self / greedy_obj / greedy_multi / greedy_none` — no `time`, `backstop`, or `suppressed` entries, confirming the high-threshold override neutralizes the time/CBR paths.
- `GreedyBSMImplied_v2` (20 s): `num_objects_sent = 0` in every row of `-tx.csv`, header-only payload confirmed. Scheduler behaves identically to Greedy_v2.
- `HybridSDSM_v2` (20 s): regression — still emits `hybrid_*` prefixed reasons, spatial association counters populated (`assoc_parsed ≈ 3.7 M`, `heading_fallback = 22` over 20 s), object-AoI sampler active. The helper extraction did not regress any v2 behavior.
- `Greedy` (v1, 20 s): summary reports `assoc_*` counters as 0 and `object_aoi` sentinel `-1`; `-triggers.csv` / `-object-aoi.csv` exist as header-only files (opened statically but never written because the write gates are v2-only). Algorithmic behavior and numeric output are unchanged.

### Why this split was done this way

The original brief explicitly said *"Do not remove or alter the Greedy, GreedyBSMImplied, EventTriggered, or Periodic baselines. The rerun needs them for comparison."* That instruction is respected: all v1 configs (`[Config Greedy]`, `[Config EventTriggered]`, `[Config GreedyBSMImplied]`, `[Config HybridSDSM]`) are untouched, so the archived v1 numbers remain reproducible. The new v2-scheduler variants are additive — opt-in via `greedyVariant="v2"` — so a user can choose to compare literature-consistent schedulers across Greedy-family + HybridSDSM in one rerun, or stick with the original v1 baseline mix.
